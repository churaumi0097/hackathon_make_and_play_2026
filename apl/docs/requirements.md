## 要件定義
## 機能要件
1. 感情入力
プリセットボタン（後悔・歓喜・焦燥 など）による感情選択。
フリーテキストによる「泥臭い感情」の入力フォーム。
遠回り強度ダイヤル（少しだけ / しっかり / とことん）を選択できる。
（Phase2）音声入力。
受け入れ条件：感情が未入力でもデフォルト（ニュートラル）でルート生成できる。


2. LLM連携（Gemini API 等）
入力テキストから感情ベクトル（valence × arousal）を解析する。
リザルト用の「皮肉と労いの効いたメッセージ」を生成する。
受け入れ条件：LLM応答が失敗しても、プリセット感情のみでフォールバック動作する。


3. ルート生成・描画
現在地と目的地を取得する。
感情ベクトル → ルートアーキタイプ → 経由POIを算出（仕様は emotion-routing.md）。
遠回り強度と安全要件の範囲内でルートを描画する。


4. ナビゲーション
生成ルートに沿って現在地をトラッキングする簡易マップ表示。
「最短で帰る」ボタンを常時表示（1タップで通常ナビに切替）。


5. リザルト（ゴール）
目的地到着を判定する。
「本来の最短時間」と「実際の所要時間」の差分を算出する。

表示は「無駄にした時間」ではなく 「感情に使った時間」 としてポジティブに提示する。

到着時の感情を再測定し、出発時からの変化（before → after）を表示する。
通過したPOIを“拾いもの”として一覧化する。
SNSシェア用リザルトカード（画像）を生成する。（Phase2で画像生成、Phase1はテキスト共有可）


## 非機能要件

1. レスポンス
LLMの待ち時間はローディング演出で飽きさせない（例：「AIの最適解を破壊中…」）。
ルート生成の目標応答：おおむね数秒以内。超過時は演出を継続。


2. UI/UX
一般的なナビの「無機質な白と青」を避け、ノスタルジックなトーン・人間味のあるフォントと配色。
夜間・片手操作を想定し、タップ領域を大きく、暗所で見やすい配色にする。


3. プライバシー
感情テキスト・位置情報は機微データ。原則ローカル保持とし、サーバ永続化する場合は同意を取る。
LLM送信は必要最小限に留める。


## 安全要件（最優先）
夜間は、治安の悪いエリア・暗い道・物理的危険箇所を経由地から除外する。
歩道のある道を優先し、幹線道路の路肩など危険な区間を避ける。
遠回りの総距離・総時間に上限を設ける（特に夜間・悪天候時は上限を厳しくする）。
悪天候時は水辺など危険なPOIをルーティングしない。
ユーザーはいつでも1タップで最短帰宅に切り替えられる。
